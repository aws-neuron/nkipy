## What the dumps show — the dominant waste is per-op HBM round-trips

Across every pattern the compute ops are a small minority; most nki ops are
`alloc` + `dma_copy` + `dealloc` + `view` scaffolding around each op. The
direct-lower pipeline puts an **HBM boundary between segments**, so a chain like
rmsnorm's `mul(x,x) -> reduce -> add -> sqrt -> broadcast -> div -> mul` becomes
several segments, each doing: load inputs from HBM -> 1 compute -> store result
to HBM. The intermediate never stays on-chip even though the next op consumes it
immediately.

Concrete in `rmsnorm.nki`:
- `mul(x,x)` loads `x`, computes, **stores the (128,1024) square back to HBM**.
- the reduce segment **reloads** that same tensor from HBM to reduce it.
- 9 compute ops, but 23 DMAs / 34 allocs / 22 deallocs / 21 views.

Even the parity case `swiglu_act.nki` stores an intermediate (`%t12`, the silu
result) to HBM that is only consumed by the next multiply, plus a redundant
final load+store to copy into the output buffer.

## Implication

The biggest op-count reduction is **cross-op SBUF residency / fusion**: keep a
segment's result on-chip when the next op is its consumer, instead of
store-to-HBM + reload. This directly removes the load+store+alloc+dealloc per
boundary (the ~88% of ops that aren't compute). HLO's BIR does exactly this —
its softmax keeps the whole max/sub/exp/sum/div chain resident (Traffic =
input + output only). See `../../../../nkigen-lite/PERFORMANCE.md`.
